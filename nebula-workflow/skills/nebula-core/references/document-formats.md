# .planning/ 文档格式规范

本文档定义星云工作流 Plugin 使用的所有文档格式。
完全兼容 GSD 生成的 `.planning/` 文档。

## 目录结构

```
.planning/
├── PROJECT.md            # 项目背景文档（必需）
├── REQUIREMENTS.md       # 需求文档（必需）
├── ROADMAP.md            # 路线图文档（必需）
├── STATE.md              # 项目状态文档（必需）
├── config.json           # 工作流配置（必需）
├── research/             # 技术研究资料（可选）
│   └── *.md
├── phases/               # 阶段目录
│   ├── 1/
│   │   ├── CONTEXT.md    # 讨论上下文（可选）
│   │   ├── RESEARCH.md   # 技术研究（可选）
│   │   ├── PLAN.md       # 执行计划（必需）
│   │   ├── PLAN-REVIEW.md # 规划审核报告（由 nx-check-plan 生成）
│   │   ├── VALIDATION.md # 代码验证报告（由 nx-validate 生成）
│   │   ├── VERIFICATION.md # 目标验证报告（由 nx-verify 生成）
│   │   └── PHASE-SUMMARY.md # 执行摘要（由 nx-exec 生成，记录偏差与存根）
│   ├── 2/
│   │   └── ...
│   └── ...
├── quick/                # 快捷记录（可选）
├── intel/                # 代码库分析记录（可选）
└── PAUSE.md              # 暂停状态记录（由 /nx-pause 生成，/nx-resume 读取，完成阶段后由 /nx-done 清理）
```

---

## PROJECT.md

项目背景文档，描述项目的整体目标和约束。

```markdown
# [项目名称]

## What This Is

[2-3 句话描述产品做什么、给谁用]

## Core Value

[一句话核心价值 — 如果其他都失败了，这个必须成功]

## Requirements

### Verified（已验证）

<!-- 已交付并验证的需求 -->

- ✓ [需求] — [阶段/版本引用]

### Active（活跃）

<!-- 当前范围，正在构建 -->

- [ ] [需求 1]
- [ ] [需求 2]

### Out of Scope（范围外）

- [排除项 1] — [原因]
- [排除项 2] — [原因]

## Context

[背景信息：技术环境、相关工作、用户反馈]

## Constraints

- **[类型]**：[内容] — [原因]
- **[类型]**：[内容] — [原因]

类型包括：技术栈、时间线、预算、依赖、兼容性、性能、安全

## Key Decisions

| 决策 | 理由 | 结果 |
|------|------|------|
| [选择] | [为什么] | [✓ 好 / ⚠️ 需回顾 / — 待定] |

---

*最后更新：[日期] 在 [触发条件] 之后*
```

## REQUIREMENTS.md

```markdown
# 需求文档

## 已验证

<!-- 已交付并验证的需求列表 -->

- ✓ [需求 1] — [实现于阶段 X]
- ✓ [需求 2] — [实现于阶段 Y]

## 活跃

<!-- 当前正在构建或待构建 -->

- [ ] [需求 3] — [描述]
- [ ] [需求 4] — [描述]

## 范围外

<!-- 明确的边界 -->

- [排除项] — [原因]
```

## PLAN.md（阶段执行计划）

阶段执行计划，包含任务分解、依赖分析、wave 分组。

```markdown
# 阶段 [N] 执行计划

## 阶段信息

**路线图引用**：[来自 ROADMAP.md]
**复杂度**：S/M/L
**状态**：待开始 | 执行中 | 已完成
**预估工作量**：[人天]

## Must-Haves（可验证目标）

<!-- 从阶段目标推导的可观测真相、期望制品和关键连线。
     plan-checker 据此审核计划完整性，verifier 据此验证目标达成。 -->

**可观测真相**（truths）：如果这些为真，阶段目标基本达成
- [用户可观测的行为描述，如 "用户能发送文本消息"]
- [用户可观测的行为描述，如 "发送失败时返回错误提示"]

**期望制品**（artifacts）：
| 路径 | 提供的能力 | 最小规模 |
|------|-----------|---------|
| `src/api/MessageController.kt` | 消息发送 HTTP 端点 | ~40 行 |
| `src/service/MessageService.kt` | 消息业务逻辑 | ~60 行 |

**关键连线**（key_links）：制品间必须存在的连接
| 从 | 到 | 方式 |
|----|-----|------|
| MessageController | MessageService | POST 路由调用 sendMessage() |
| MessageService | MessageRepository | 调用 save() 持久化 |
| MessageHandler | MessageService | 通过注入调用 |

## 任务分解

### Wave 1：基础设施
<!-- 无依赖任务，可并行 -->

#### 任务 1.1：[任务名]
**状态**：待开始 | 执行中 | 已完成 | 失败
**前置**：无
**描述**：[需要做什么]
**验收标准**：
- [ ] [标准 1]
- [ ] [标准 2]
**关联文件**：[文件路径]
**复杂度**：S/M/L
**结果**：[执行完成后记录]

### Wave 2：核心逻辑

#### 任务 2.1：[任务名]
**状态**：待开始
**前置**：任务 1.1
**描述**：[需要做什么]

### Wave 3：集成与验证

## WAVE 配置

waves:
  1:
    tasks: [1.1, 1.2]
  2:
    tasks: [2.1, 2.2]
  3:
    tasks: [3.1]

## 依赖图

```
任务 1.1 ──→ 任务 2.1 ──→ 任务 3.1
任务 1.2 ──→ 任务 2.2
```

## Gap Closure（修复计划）

#### 任务 1.G1：[修复描述]
**状态**：待开始
**gap_closure**: true
**描述**：[修复验证发现的问题]
```

## PAUSE.md（暂停状态）

工作流暂停时的状态快照，由 `/nx-pause` 生成，`/nx-resume` 读取。完成阶段后由 `/nx-done` 清理。

```markdown
# 工作流暂停状态

## 当前状态

- **阶段**：[阶段号]
- **当前 Wave**：[1/2/3 或 未开始]
- **暂停时间**：[ISO 日期时间]

## 任务状态

| 任务 | 状态 | 备注 |
|------|------|------|
| [编号] | 已完成/执行中/待开始/失败 | [进度说明] |
| [编号] | 已完成/执行中/待开始/失败 | [进度说明] |

## 暂停原因

[用户备注或"用户主动暂停"]

## 恢复指引

执行 `/nx-resume` 恢复工作流，将自动从阶段 [N] 的任务 [X.Y] 继续。
```

## TDD 计划格式

当阶段在 ROADMAP.md 中标注 `类型: tdd` 时，PLAN.md 中的实现任务需追加 TDD 字段。

### 阶段级配置（ROADMAP.md）

在 ROADMAP.md 的概览表中追加 `类型` 列：

```markdown
| 阶段 | 名称 | 复杂度 | 状态 | 类型 |
|------|------|--------|------|------|
| 5 | 用户认证 | M | 待开始 | tdd |
```

### 任务级标注（PLAN.md 中的 TDD 任务格式）

```markdown
#### 任务 5.1：实现密码哈希
**状态**：待开始
**类型**：tdd
**TDD 阶段**：RED | GREEN | REFACTOR
**RED 验收标准**：
- [ ] 测试文件存在且编译通过
- [ ] 测试在实现前验证失败（预期行为）
- [ ] git log 中存在 `test(5-1):` 提交
**GREEN 验收标准**：
- [ ] 最小实现使所有测试通过
- [ ] git log 中存在 `feat(5-1):` 提交
**REFACTOR 验收标准**（可选）：
- [ ] 重构后所有测试仍通过
- [ ] git log 中存在 `refactor(5-1):` 提交（如有变更）
**关联文件**：`src/main/kotlin/.../PasswordHasher.kt`、`src/test/kotlin/.../PasswordHasherTest.kt`
**复杂度**：M
```

### TDD 提交规范

| 阶段 | 提交前缀 | 示例 |
|------|---------|------|
| RED | `test(<阶段号>-<任务号>):` | `test(5-1): 密码哈希函数测试用例` |
| GREEN | `feat(<阶段号>-<任务号>):` | `feat(5-1): 实现 bcrypt 密码哈希` |
| REFACTOR | `refactor(<阶段号>-<任务号>):` | `refactor(5-1): 提取 Salt 生成逻辑` |

### TDD 任务选择启发式

- **适合 TDD**：具有明确输入/输出的业务逻辑、API 端点、数据转换、验证规则、算法、状态机
- **不适合 TDD**：UI 布局/样式、配置变更、胶水代码、一次性脚本、简单 CRUD、探索性原型
- **判断标准**：能否在编写 `fn` 之前写出 `expect(fn(input)).toBe(output)`？若能，则适合 TDD

# /nx-discuss — 讨论阶段

## 目的

分析阶段边界内的灰色区域，逐一提问引导用户做决策，生成 CONTEXT.md 供后续规划使用。
遵循"用户是愿景者，你是建造者"原则 — 收集决策，不做实现。

## 流程

### 第一步：加载上下文

1. 读取 `.planning/ROADMAP.md`，确认指定阶段的描述
   - 如果没有参数，列出所有未规划阶段让用户选择
   - 验证阶段号有效

2. 读取阶段详情（如 `.planning/phases/<N>/` 下已有文件）
   - 如果已有 CONTEXT.md，询问是否重新讨论或继续

3. 读取 `.planning/PROJECT.md` 了解项目全局约束

### 第二步：分析灰色区域

从 ROADMAP.md 的阶段描述和 PROJECT.md 的约束中，识别需要决策的问题：

**典型问题类型：**
- 技术选型（如有多个合理选项）
- 边界条件（正常/空/错误状态如何处理）
- 行为偏好（用户期望的具体表现）
- 优先级权衡（多个需求冲突时的取舍）
- 设计风格（如果涉及 UI）

**每个问题附带选项和优缺点**（遵循记忆中的【决策选项需附带优缺点说明】原则）。

### 第三步：逐一提问决策

对每个识别出的问题，使用 `AskUserQuestion` 引导用户决策：

**提问技巧：**
- 不要一次问太多（最多 3 个一组）
- 每个问题提供 2-3 个具体选项
- 标注推荐选项
- 等待用户回答后再继续下一组

**不需要问的问题（自己可以判断）：**
- 代码库已有明确模式的问题 → 直接读代码推断
- 技术层面的常规实现方式 → 留给 researcher 和 planner
- 标准框架使用方式 → 不需要讨论

### 第四步：生成 CONTEXT.md

将讨论结果写入 `.planning/phases/<N>/CONTEXT.md`：

```markdown
# 阶段 <N> 讨论上下文

## 阶段范围
[来自 ROADMAP.md 的阶段描述]

## 锁定决策
| 决策 | 选择 | 原因 |
|------|------|------|
| ...  | ...  | ...  |

## 用户偏好
- [具体偏好 1]
- [具体偏好 2]

## 排除项
- [明确排除的内容]

## 注意事项
- [来自用户的特殊提醒]
```

### 第五步：结果输出

显示讨论总结：
```markdown
## 讨论完成 — 阶段 <N>

锁定决策数：X
排除项：Y

下一步：执行 `/nx-plan <N>` 开始规划此阶段
如果不需要讨论，可直接 `/nx-plan <N> --skip-discuss`
```

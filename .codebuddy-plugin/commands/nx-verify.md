# /nx-verify — 验证工作

## 目的

逐条检查 PLAN.md 中的任务验收标准是否达成。
发现问题则生成修复计划（以 gap_closure 标记的任务）。
全部通过则更新 STATE.md。

## 前置条件

- `.planning/phases/<N>/PLAN.md` 必须存在
- 建议先执行 `/nx-exec <N>` 运行过实现

## 流程

### 第一步：加载验证上下文

1. 读取 `.planning/phases/<N>/PLAN.md`
2. 识别所有任务和其验收标准
3. 读取 `.planning/PROJECT.md` 了解全局约束

### 第二步：逐项验证

对每个标记为"已完成"的任务，逐条检查验收标准：

**验证方法：**
- 文件存在性检查：`Glob`/`test -f` 验证关联文件是否存在
- 代码检查：`Grep`/`Read` 验证关键实现是否存在
- 接口检查：验证公开 API/接口是否按计划实现
- 行为检查：通过描述判断是否符合预期

**验证结果标记：**
- ✅ 通过 — 验收标准全部满足
- ⚠️ 部分通过 — 基本功能实现了，但有些细节需修复
- ❌ 未通过 — 功能未实现或严重偏离计划

### 第三步：报告结果

输出验证报告：

```markdown
## 验证报告 — 阶段 <N>

### 通过 (X)
- ✅ 任务 X.Y — [验收标准全部满足]
- ✅ 任务 X.Z — [验收标准全部满足]

### 部分通过 (Y)
- ⚠️ 任务 X.W — [缺失: 错误处理、边界条件]
  修复建议：[具体建议]

### 未通过 (Z)
- ❌ 任务 X.V — [功能未实现]
  修复建议：[具体建议]
```

### 第四步：生成修复计划（如果有问题）

如果有 ⚠️ 或 ❌ 的任务：
1. 在 PLAN.md 中追加 gap_closure 任务：

```markdown
### 任务 X.G1: [修复任务]
**状态**: 待开始
**gap_closure**: true
**描述**: 修复验证发现的问题
**验收标准**:
- [ ] 修复问题 1
- [ ] 修复问题 2
- [ ] 验证原始验收标准通过
```

2. 询问用户是否立即修复：
   - 是 → 提示执行 `/nx-exec <N> --gaps-only`
   - 否 → 记录到 STATE.md 作为未完成项

### 第五步：全部通过

如果所有任务验证通过：
1. 在 STATE.md 中更新当前阶段验证状态
2. 输出：

```markdown
## ✅ 验证通过 — 阶段 <N>

所有 X 个任务验收标准均已满足。

下一步：执行 `/nx-done <N>` 完成此阶段
```

# 阶段讨论

## 目标
为阶段 N 加载已有上下文、识别需要决策的灰区、通过交互式问答收集决策，最终生成 `NN-CONTEXT.md`。

当前实现：仅 **default 模式**（全量灰区讨论）。未来可扩展多模式（power/auto/batch 等）。

## 参数
- `$ARGUMENTS`：阶段编号 N（必需）

## 门禁检查

### Pre-flight Gate
```bash
# 检查 PROJECT.md 和 ROADMAP.md 存在
if [ ! -f ".planning/PROJECT.md" ] || [ ! -f ".planning/ROADMAP.md" ]; then
  报错: "PROJECT.md 或 ROADMAP.md 缺失，请先执行 /nx-init"
fi
```

### Closed-Phase Gate
```bash
# 检查阶段是否已完成（禁止对已完成阶段重新讨论）
PHASE_STATUS=$(grep "| $PHASE_NUM " .planning/STATE.md | awk -F'|' '{print $2}' | tr -d ' ')
if [ "$PHASE_STATUS" = "Complete" ]; then
  警告: "阶段 N 已完成，重新讨论可能覆盖已有决策。使用 /nx-discuss N --force 强制执行。"
fi
```

## 流程

### 步骤 1：加载已有上下文
读取以下文件获取已有信息：
1. `.planning/PROJECT.md` —— 项目定义和技术约束
2. `.planning/REQUIREMENTS.md` —— 本阶段关联的需求
3. `.planning/ROADMAP.md` —— 阶段目标和依赖
4. `.planning/STATE.md` —— 当前进度
5. 已完成阶段的 `NN-CONTEXT.md` —— 已有决策（避免重复提问）

### 步骤 2：分析灰区
识别以下类型的未决问题：
- **技术选择**: 库/框架/模式的选择
- **接口设计**: API 契约、数据格式
- **架构决策**: 模块划分、依赖方向
- **边界条件**: 错误处理、并发控制、性能目标
- **实现策略**: 增量方式、测试策略

**覆盖度自检**（分析完成后执行，内部审计，不自动追加灰区）：

对照以下核心维度，检查已识别的灰区是否覆盖：

| 维度 | 含义 | 适用场景 |
|------|------|---------|
| 行为/交互 | 用户操作后的响应、状态转换、边界条件 | 所有阶段 |
| 数据模型/存储 | 数据 schema、持久化策略、一致性要求 | 涉及数据时 |
| 错误/异常处理 | 失败模式、降级策略、错误反馈 | 涉及 I/O 或外部服务时 |
| 性能/规模 | 预期负载、延迟要求、缓存策略 | 涉及列表/搜索/实时时 |
| 安全/权限 | 访问控制、数据可见性、操作审计 | 涉及用户数据或敏感操作时 |
| 视觉/呈现 | 布局、样式惯例、响应式策略 | 涉及 UI 时 |
| 集成/外部依赖 | 第三方 API、协议选择、版本锁定 | 涉及外部服务时 |

**判定规则：**
- 某维度被"覆盖" = 至少有一个灰区直接处理该维度
- 0-1 个缺口 → 分析充分，不做额外操作
- 2+ 个缺口 → 记录为 `<覆盖缺口>`（维度名 + 为什么可能重要），留待步骤 3 的 "继续探索" 提示使用
- `<覆盖缺口>` 仅用做提示源 — 不要为了填补缺口而编造灰区

### 步骤 3：交互式讨论

**讨论流程：**
1. 呈现所有已识别的灰区，让用户选择优先讨论哪些（multiSelect）
2. 逐项讨论选中的灰区，每项提供选项（含优缺点），收集用户决策
3. 全部选中项讨论完毕后，进入"继续探索"提示

**继续探索提示（改进版）：**
- 检查步骤 2 的 `<覆盖缺口>`：
  - 有缺口 → 主动列出为候选选项，例如：
    ```
    ☐ 错误/异常处理 — 讨论失败降级和错误反馈
    ☐ 性能/规模 — 讨论预期负载和缓存策略
    ```
  - 无缺口 → 回退到通用 "探索更多灰区"
- 总是提供 "讨论完成，生成 CONTEXT.md" 选项
- 用户选择缺口选项后 → 将维度翻译为 1-2 个针对本阶段的具体问题 → 回到逐项讨论流程
- 循环直到用户选择 "讨论完成"
- 记录讨论过程中的关键观点到 `NN-DISCUSSION-LOG.md`

### 步骤 4：生成 CONTEXT.md

```markdown
---
phase: N
status: contexted
---
# Phase N: <阶段名称> — 上下文

## 阶段目标
（从 ROADMAP.md 提取）

## 关联需求
（从 REQUIREMENTS.md 提取）

## 技术决策
（从讨论中提取的已锁定决策，标注 D-编号）

## 实现约束
（PROJECT.md 约束 + 本阶段特殊约束）

## 灰区已解决
（本次讨论解决的问题列表）

## 灰区遗留
（本次讨论未解决的问题及其原因）
```

## 成功标准
- 已有上下文已加载且应用（无重复提问）
- 灰区已识别并通过智能分析分类
- 用户选择了要讨论的灰区范围
- 每个选定灰区已充分探索
- CONTEXT.md 捕获的是决策，而非模糊愿景
- 用户了解后续步骤（/nx-plan N）

# 深度审核（深度二审）

## 目标
对阶段 N 的 PLAN.md 进行深度质量审核。与 `/nx-plan` 内部的基础门禁审核不同，深度二审聚焦于基础审核不覆盖的领域：

- **跨计划契约**：多个 PLAN 之间的接口契约是否一致
- **长尾风险**：边界条件、并发场景、异常路径的技术风险
- **历史阶段偏离**：与已完成阶段建立的代码模式是否偏离
- **停滞检测**：如果连续审核问题数不减少，提前升级

支持独立调用（不依赖完整 plan 流程），也可在基础门禁审核通过后进行补充审查。

## 参数
- `$ARGUMENTS`：阶段编号 N（必需）

## 门禁检查

### Pre-flight Gate
```bash
# 检查 PLAN.md 存在
PLAN_COUNT=$(find .planning/phases/0${N}-* -name "0${N}-*-PLAN.md" | wc -l | tr -d ' ')
if [ "$PLAN_COUNT" -eq 0 ]; then
  报错: "阶段 N 没有 PLAN.md 文件，请先执行 /nx-plan N"
fi
```

## 审核维度

### 0. 复核基础门禁（如果已由 nx-plan 审核）
- [ ] 所有 ROADMAP.md 中的阶段需求是否被覆盖
- [ ] 每个任务是否有验证方法和验收标准

### 1. 跨计划契约检查
- [ ] 不同 PLAN 之间的接口契约是否一致
- [ ] Wave 间依赖的接口签名是否匹配
- [ ] 共享模块（Entity/Config）的修改是否所有消费方已知

### 2. 长尾风险分析
- [ ] 边界条件：空输入、大数据量、超时等极端场景
- [ ] 并发场景：多请求竞争、事务边界
- [ ] 异常路径：网络失败、DB 不可用、依赖超时
- [ ] 是否有未被识别的技术风险

### 3. 历史阶段偏离审查
- [ ] 与已完成阶段的代码模式是否一致（对比 PATTERNS.md）
- [ ] 命名规范、目录结构是否符合项目约定
- [ ] 是否有破坏性变更需要特殊处理（对比已发布的 API/Entity）

### 4. 停滞检测
追踪连续审核迭代的问题数量：
```
第 N 次审核：问题数 = X
第 N+1 次审核：问题数 = Y
如果 Y >= X（问题数未减少）→ 提前升级到用户决策
max 迭代次数 = 3
```

## 输出格式

```markdown
## 审核结果：阶段 N

### 审核摘要
- 审核次数：M/N（max 3）
- 审核状态：PASSED / ISSUES FOUND
- 问题数：X（阻塞 Y / 警告 Z）

### 问题列表
| # | 严重度 | 类别 | 描述 | 建议修复 |
|---|--------|------|------|---------|

### 最终裁决
- [ ] APPROVED —— 可以执行
- [ ] REVISION NEEDED —— 需要修改后重新审核
- [ ] ESCALATED —— 需要用户决策

## CHECK-PLAN COMPLETE
```

## 成功标准
- 所有阻塞问题已解决或升级
- 警告问题已记录（不阻塞执行）
- 审核结果已写入 `NN-PLAN-CHECK.md`

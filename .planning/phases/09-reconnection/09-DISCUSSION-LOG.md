# Phase 9: Reconnection — 讨论日志

## 会话信息
- 日期: 2026-06-13
- 参与者: 开发者

## 灰区列表

### 1. 重连状态机 — 服务器端 vs 客户端实现
- **描述**: 重连状态机（INITIAL → BACKOFF → CONNECTING → CONNECTED）的归属问题
- **选项**:
  - A: 纯客户端实现，服务端仅提供无状态 Token 验证（登录复用）
  - B: 服务端维护重连状态（跟踪重连次数、状态）
  - C: 服务端提供重连专用 RPC（简化客户端逻辑）
- **影响**: 决定了 RECON-01 的实现位置和接口设计

### 2. 心跳恢复策略
- **描述**: 重连后心跳机制的恢复时机和参数
- **选项**:
  - A: 重连成功后立即恢复 gRPC keepalive，应用层心跳由客户端发起后恢复
  - B: 重连后等待首次 PONG 响应后再恢复完整心跳
  - C: 重连后走快速心跳（缩短间隔）3 次后恢复标准间隔
- **影响**: 涉及 RECON-02 心跳恢复的实现细节

### 3. 消息优先级分级策略
- **描述**: P0/P1 消息的判定标准和重试策略
- **选项**:
  - A: 按消息类型分级（聊天消息=P0，系统通知=P1）
  - B: 按发送时间分级（最近 5 分钟=P0，更早=P1）
  - C: 客户端指定优先级（消息 proto 增加 priority 字段）
- **影响**: 决定 RECON-03 的实现和 proto 是否需要扩展

### 4. 断连期间消息积压策略
- **描述**: 用户断连期间服务端如何处理发送给该用户的消息
- **选项**:
  - A: 继续 Redis 队列写入，重连后统一拉取（复用 Phase 6 的离线消息机制）
  - B: 仅在用户在线时推送，断连期间丢弃
  - C: 断连期间缓存到本地队列，超过容量后丢弃最旧
- **影响**: 与 Phase 3/6 的离线消息机制耦合

### 5. 原子连接清理 — Redis Pipeline 实现细节
- **描述**: RECON-04 的旧连接清理的原子性保证
- **选项**:
  - A: Redis pipeline（非事务）批量操作，最终一致即可
  - B: Redis MULTI/EXEC 事务保证原子性
  - C: Lua 脚本确保原子性（最佳性能）
- **影响**: 涉及 RECON-04 的实现可靠性和性能

### 6. 最大重试次数与重置时机
- **描述**: 重试次数的计数规则和重置条件
- **选项**:
  - A: 固定次数（如 5 次）后停止，用户手动重试
  - B: 滑动窗口（如 1 小时内最多 10 次）
  - C: 无限制重试，每次重试间隔指数增长
- **影响**: 涉及 RECON-02 的防滥用保护

### 7. 连接迁移期间的消息暂存
- **描述**: 新连接建立但旧连接尚未清理时的消息路由
- **选项**:
  - A: 新连接建立后立即切换路由，旧连接消息继续处理直到清理
  - B: 新连接建立后缓存消息，待旧连接清理后开始投递
  - C: 新连接建立时立即关闭旧连接（先断后连）
- **影响**: 消息不丢失的保障

### 8. 服务端主动通知客户端断连
- **描述**: 服务端检测到连接异常时是否主动通知客户端
- **选项**:
  - A: 不主动通知，客户端通过心跳超时自行检测
  - B: 服务端通过 gRPC Status 返回特定错误码
  - C: 服务端在旧 Channel 关闭前推送 DISCONNECT 通知
- **影响**: 客户端重连触发策略

### 9. 伪在线状态与重连的关系
- **描述**: 阶段 8 实现的 60s 伪在线与重连状态机的关系
- **选项**:
  - A: 伪在线独立运行，60s 超时后才进入重连逻辑
  - B: 伪在线是重连状态机的一部分，伪在线期间即为 BACKOFF 状态
  - C: 伪在线和重连并行，用户在线状态先于重连逻辑展示
- **影响**: 与 Phase 8 在线状态的集成

## 讨论记录

### 灰区 1：重连状态机归属
- **选择**: A. 纯客户端实现
- **理由**: 服务端无需额外状态，复用现有 login Token 验证即可
- **决策**: D-61

### 灰区 2：心跳恢复策略
- **选择**: B. 等待首次 PONG
- **理由**: 确认双向通信正常后再恢复标准心跳，更可靠
- **决策**: D-62

### 灰区 3：消息优先级分级
- **选择**: C. 客户端指定优先级
- **理由**: 客户端本地维护 P0/P1 重试队列，服务端无需感知优先级，无需改动 proto
- **决策**: D-63

### 灰区 4：断连期间消息积压
- **选择**: A. 复用离线消息
- **理由**: 利用现有 Redis 离线消息队列，重连后 message/pull 拉取，零新开发量
- **决策**: D-64

### 灰区 5：原子连接清理
- **选择**: A. Redis pipeline
- **理由**: 性能最佳，最终一致性即可接受
- **决策**: D-65

### 灰区 6：最大重试次数
- **选择**: B. 滑动窗口
- **理由**: 1h 内最多 10 次，指数增长间隔（2s→4s→8s→…→上限 60s）
- **决策**: D-66

### 灰区 7：连接迁移消息暂存
- **选择**: B. 缓存再投递
- **理由**: 确保消息不丢失，旧连接清理完成后再投递到新连接
- **决策**: D-67

### 灰区 8：服务端主动通知断连
- **选择**: C. 推送 DISCONNECT 通知
- **理由**: 客户端即时感知，可提前进入重连流程
- **决策**: D-68

### 灰区 9：伪在线与重连的关系
- **选择**: B. 伪在线 = BACKOFF
- **理由**: 状态机与在线状态同步，60s 超时后标记离线
- **决策**: D-69

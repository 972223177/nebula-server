# Plan 10-02: 三态跟踪与 DeliveryAck 推送

## 目标

实现消息交付三态跟踪（sent → delivered → read），新增 `DeliveryTrackingService` 封装 Redis Hash 状态变更逻辑，扩展 `PushService` 在推送成功后触发 delivered 标记并推送 `DELIVERY_ACK` 给发送者，修改 `MessageService` 在消息发送时写入序列号。状态模型：Redis Hash `msg:{msg_id}:delivery`，字段 `{uid}:status`，每次状态变更刷新 TTL（7 天）。

## 验证

1. **编译验证**：`./gradlew build` — 全量编译通过
2. **单元测试**：
   - `DeliveryTrackingServiceTest`：状态转换正确性、边界条件（多设备、跳跃转换）
   - `PushServiceTest`：DELIVERY_ACK Envelope 结构正确
   - `MessageServiceTest`：seq 自增连续性
3. **集成测试**（可选）：启动 Redis，验证 Hash 写入和读取
4. **Redis 命令验证**：`HSET msg:1:delivery uid:1 0` + `EXPIRE msg:1:delivery 604800` 手动测试

## 成功标准

1. `DeliveryTrackingService` 支持三态标记，所有边界场景（多设备重复、sent→read 跳跃）正确
2. `PushService.pushMessage()` 推送成功后自动调用 `markDelivered()`
3. `PushService.pushDeliveryAck()` 能向发送者推送 `DELIVERY_ACK` 事件
4. 消息发送响应中包含递增的 `seq` 字段
5. 状态变更时自动刷新 Redis Hash TTL 为 7 天

## 风险

- **多设备重复 delivered**：已通过 `HSETNX` 加 `GET` 预检解决
- **TTL 原子性问题**：Redis 不支持单 field TTL，整体 EXPIRE 方案在群聊场景中部分用户状态被覆盖时影响可控
- **去重下沉回退风险**：移除 `DedupStep` 中的 SETNX 后，10-04 完成前 MySQL 唯一索引是唯一防重屏障；如果 10-04 延迟或失败，建议 10-02 暂时保留 Handler 层的 SETNX，在 10-04 集成时再最终移除中间状态

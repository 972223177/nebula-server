# Plan 10-04: 死信表、补偿任务、Admin API 与 DI 集成

## 目标

实现完整的死信处理链路：JPA Entity 映射死信表、JPA Repository 提供查询、`DeadLetterService` 处理死信创建和重试补偿逻辑、10 分钟间隔的协程补偿任务、两个 Admin API Handler（分页查询 + 手动重试）、以及所有新组件的 Koin DI 注册和 HandlerCollector 集成。本 Plan 将 Phase 10 的所有新组件串联起来。

## 成功标准

1. `DeadLetterService` 支持创建死信、分页查询、自动补偿重试、手动重试
2. `DeadLetterCompensator` 每 10 分钟自动扫描并补偿 pending 死信
3. `admin/dead-letters` 和 `admin/retry-dead-letter` 可通过 gRPC 双向流调用（无需用户 Token）
4. 所有组件注册到 Koin 容器，`GatewayModule.kt` 和 `RepositoryModuleInitializer.kt` 正确集成
5. pendingBuffer 投递失败 10 次后自动创建死信记录（D-75）
6. `MessageService.sendMessage()` 中的 seq 生成统一通过 `SeqService.nextSeq()` 实现
7. `ChatHandlerModule` 不再包含 Phase 10 的交付跟踪组件，无重复 Koin 注册
8. AuthInterceptor 放行所有 `admin/` 前缀方法，无需 Token 即可通过拦截器（生产部署前需评估增加 Admin Token 校验）
9. 全量构建 `./gradlew build` 通过，无编译错误

## 风险

- **pendingBuffer 与死信集成风险**：`ChatService.ChatStreamObserver` 的 per-message 重试计数器需要小心设计，避免内存泄漏（Map 持续增长）。建议使用 `maxEntries` 限制或结合 Redis 外部计数器
- **DI 迁移冲突**：从 `ChatHandlerModule` 移除组件时需确保 10-04 执行时 10-02 的中间状态已清理干净。建议在 10-04 开始时先移除旧注册再添加新注册

# Plan 10-01: Proto 扩展与 Flyway V4 死信表迁移

## 目标

为 Phase 10（消息可靠性）所需的交付回执、序列号间隙检测和管理 API 添加 Protobuf 消息定义，同时创建死信表的数据库迁移文件。本 Plan 是后续三个 Plan（10-02 三态跟踪、10-03 间隙检测、10-04 死信补偿）的共同依赖基础。

## 任务

| # | 类型 | 文件 | 操作 | 验证 | 验收标准 |
|---|------|------|------|------|---------|
| 1 | modify | `proto/src/main/proto/nebula/chat/chat.proto` | 在 `SendMessageResp` 中追加 `int64 seq = 3` 字段（D-74：服务端分配的会话序列号） | 编译通过，生成的 Java 类包含 `getSeq()` 方法 | `SendMessageResp` 包含 `seq` 字段，编译产物中可访问 |
| 2 | create | `proto/src/main/proto/nebula/message/message.proto` | 在文件末尾（`ReadReceiptPayload` 之后）新增 `DeliveryAckPayload` 消息：`int64 msg_id = 1`、`string conversation_id = 2` | 编译通过；生成的 Java 类字段齐全 | 新增消息定义与现有消息风格一致，编译无错误 |
| 3 | create | `proto/src/main/proto/nebula/message/message.proto` | 在文件末尾新增 `MessageSeqReq`（`string conversation_id = 1`）和 `MessageSeqResp`（`int64 seq = 1`）消息 | 编译通过；Req/Resp 成对出现 | 客户端可通过 `MessageSeqReq` 查询当前会话最新 seq |
| 4 | create | `proto/src/main/proto/nebula/admin.proto` | 新建 admin.proto 文件，定义 `DeadLetterQueryReq`（page/page_size/status）、`DeadLetterQueryResp`（items/total）、`DeadLetterItem`（id/msg_id/conversation_id/sender_uid/content/fail_reason/fail_count/status/created_at）、`RetryDeadLetterReq`（dead_letter_id）、`RetryDeadLetterResp`（success） | 编译通过；Java 类生成正确 | Admin Proto 消息完整，符合分页查询和手动重试场景需求 |
| 5 | create | `repository/src/main/resources/db/migration/V4__add_dead_letters.sql` | 创建 `dead_letters` 表 DDL — 字段：id（自增主键）、msg_id、conversation_id、sender_uid、message_type、content、payload（BLOB）、client_msg_id、client_ts、fail_reason、fail_count、status（pending/retrying/permanent_failed/retry_success）、**version（int 默认 0，乐观锁）**、created_at、updated_at；含 idx_status_created 和 idx_client_msg_id 索引 | Flyway 迁移成功，表结构和索引按预期创建 | `dead_letters` 表包含完整字段和索引，Flyway 校验通过 |

## Wave 分组

- **Wave 1（无依赖，可并行）**: Plan 10-01
- **Wave 2**: Plan 10-02（依赖 10-01）、Plan 10-03（依赖 10-01）

## 验证

1. **Proto 编译验证**：`./gradlew :proto:generateProto` — 生成 Java 类到 `proto/build/generated/source/proto/`
2. **Flyway 迁移验证**：`./gradlew :repository:flywayMigrate` — 确认 V4 迁移成功
3. **构建验证**：`./gradlew build` — 全量编译通过无错误
4. **类结构验证**：确认生成的 Java 类包含预期的 getter 方法

## 成功标准

1. `SendMessageResp` 包含 `int64 seq = 3` 字段，编译产物可用
2. `DeliveryAckPayload` 消息定义在 `message.proto` 中，字段齐全
3. `MessageSeqReq/Resp` 消息定义成对存在
4. `admin.proto` 包含完整的死信查询和重试消息定义
5. `V4__add_dead_letters.sql` 迁移文件部署后，数据库中存在 `dead_letters` 表及正确的索引

## 风险

- **Proto 字段编号冲突**：新增字段的编号（seq=3）需确认不与未来字段冲突；当前 `SendMessageResp` 仅有 1 和 2，无冲突风险
- **Flyway 版本号冲突**：当前最新为 V3，V4 是下一个版本号，无冲突

# Plan 10-03: 消息序列号间隙检测

## 目标

实现消息序列号生成与间隙检测能力。服务端使用 Redis INCR 为每个（会话, 用户）对维护单调递增的序列号，消息发送时分配 seq 并回填到响应中。新增 `MessageSeqHandler` 供客户端查询当前会话最新 seq，客户端通过对比本地 seq 与服务端 seq 检测间隙并触发 forward 拉取。

## 任务

| # | 类型 | 文件 | 操作 | 验证 | 验收标准 |
|---|------|------|------|------|---------|
| 1 | create | `gateway/src/main/kotlin/com/nebula/gateway/sequence/SeqService.kt` | 创建 SeqService — 封装 Redis INCR 自增逻辑，key 模式 `conv:{conv_id}:next_seq:{uid}`；提供 `nextSeq(convId, uid): Long` 获取下一序列号（INCR 前使用 GET 检查当前值，若接近 `Long.MAX_VALUE - 10000` 则重置为 1）；提供 `currentSeq(convId, uid): Long` 查询当前值（使用 GET）；含 companion object 常量 KEY_PREFIX | 单元测试验证 INCR 自增连续；重置保护逻辑触发正确；GET 查询返回正确值 | seq 从 1 开始递增；同 conv+uid 下连续无间隙；接近 Long.MAX_VALUE 时自动重置为 1 避免溢出 |

| 3 | modify | `gateway/src/main/kotlin/com/nebula/gateway/handler/chat/ChatHandlerCollector.kt` | 将 `MessageSeqHandler` 注入构造参数，在 `registerAll()` 中 `registry.register(messageSeqHandler)` | 编译通过；HandlerCollector 注册无冲突 | `MessageSeqHandler` 通过 Collector 注册到 HandlerRegistry |

## 验证

1. **编译验证**：`./gradlew build` — 全量编译通过
2. **单元测试**：
   - `SeqServiceTest`：INCR 自增连续性、并发场景下无重复、Redis 重启后的行为
   - `MessageSeqHandlerTest`：请求-响应正确性、会话解析
3. **集成测试**（可选）：启动 Redis 验证 `conv:{conv_id}:next_seq:{uid}` INCR 自增

## 成功标准

1. `SeqService.nextSeq()` 返回严格递增的序列号
2. `MessageSeqHandler` 处理 `message/seq` 请求并返回当前 seq
3. 客户端可构建间隙检测逻辑：收到 `SendMessageResp.seq` 后与本地 `lastSeq` 对比，`seq > lastSeq + 1` 时触发 `PullMessages(forward, cursor=lastSeq)`
4. 所有新组件在 DI 中可解析

## 风险

- **Redis 持久化丢失**：Redis 重启无持久化时 seq 重置为 0，客户端本地 seq 可能大于服务端 seq。此场景下客户端应全量拉取会话历史。已在 D-74 中评估为可接受风险。
- **SNOWFLAKE ID 与 seq 的区别**：seq 是 per-(conv,uid) 自增，与全局 Snowflake ID 无关，两者不冲突。

# Plan 11-02：数据一致性与竞态修复

## 目标

修复 11 个 HIGH 严重性数据一致性与竞态问题：通过 TransactionTemplate 事务包裹跨 Repository 写入、DB 唯一约束 + 幂等 catch 消除好友双向竞赛、SeqService 启动时从 MySQL 恢复序列号、memberCount JPQL 原子更新替代 loadCount→set→save。

## 风险

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| V5 migration 在已有重复数据的表上执行失败 | 迁移阻塞 | 迁移前 `SELECT from_uid, to_uid, status, COUNT(*) FROM friend_requests GROUP BY ... HAVING COUNT(*)>1` 清理重复数据 |
| `runBlocking` + `TransactionTemplate.execute` 桥接可能造成线程切换 | 事务传播异常 | 在 `runBlocking` 内不切换协程上下文；Service 方法内 `withContext(Dispatchers.IO)` 在事务包裹下需移除（事务绑定 ThreadLocal） |
| `saveAndFlush()` 性能影响 | 好友添加吞吐量微降 | 好友添加非高频操作，flush 开销可忽略（~1ms） |
| SeqService 恢复逻辑查询全量会话 | 服务启动时间略增 | 并发遍历会话（`coroutineScope { sessions.map { async { restore(it) } }.awaitAll() }`） |

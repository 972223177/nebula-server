    **追加方法 1 — 查询 PEL 待处理消息数（DB-05）：**

    ```kotlin
    /**
     * 查询消费者组的 Pending Entry List（PEL）长度。
     *
     * PEL 中存储了已投递但未确认（XACK）的消息。
     * 断线重连时，可通过此方法判断是否有离线消息待处理。
     * 返回 [Map] 包含 total 等统计数据。
     */
    suspend fun getPendingCount(): Map<String, Any>? {
        return redis.xpending(STREAM_KEY, CONSUMER_GROUP)
    }
    ```

    **追加方法 2 — 获取 PEL 中的未确认消息列表：**

    ```kotlin
    /**
     * 获取 PEL 中的未确认消息列表。用于断线重连时拉取离线消息（DB-05）。
     *
     * 返回指定范围内的未确认消息条目。
     * @param start 起始 ID（"-" 表示最早）
     * @param end 结束 ID（"+" 表示最新）
     * @param count 返回条数上限
     */
    suspend fun getPendingMessages(start: String = "-", end: String = "+", count: Long = 100): List<StreamMessage<String, String>> {
        val result = redis.xpendingRange(
            STREAM_KEY, CONSUMER_GROUP,
            Range.create(start, end),
            Limit.Builder().count(count)
        )
        return result ?: emptyList()
    }
    ```

    **追加方法 3 — 按消息 ID 从 Stream 中读取指定消息：**

    ```kotlin
    /**
     * 按消息 ID 范围从 Stream 中读取消息。
     *
     * 配合 getPendingMessages 使用：获取 PEL ID 列表后，读取消息完整内容。
     */
    suspend fun readMessagesById(startId: String, endId: String, count: Long = 100): List<StreamMessage<String, String>> {
        val result = redis.xrange(
            STREAM_KEY,
            Range.create(startId, endId),
            Limit.Builder().count(count)
        )
        return result ?: emptyList()
    }
    ```

    **新增 import：**

    - `io.lettuce.core.Limit`

    **注意：**

    - KDoc 引用 D-04, DB-05

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| User action → unread count update | Phased update via @Modifying query; uncommitted reads could show stale counts |
| Read receipt → last_read_message_id update | Atomic UPDATE; no uniqueness enforcement on last_read_message_id |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-13 | Tampering | incrementUnreadCount @Modifying | mitigate | JPQL parameterized binding prevents injection; senderId exclusion uses `<>` operator with @Param binding |
| T-03-14 | Spoofing | updateReadReceipt with arbitrary lastReadMsgId | accept | lastReadMsgId is client-provided; Phase 6 Service layer should validate that the user has access to the conversation before calling this |
| T-03-15 | Repudiation | Unread count under/over count due to concurrent updates | accept | Increment/decrement is not atomic in multi-threaded scenario; Phase 11 (perf/monitoring) should monitor drift. For v1, eventual consistency is acceptable |
</threat_model>

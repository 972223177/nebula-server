    - `class LogInterceptor : Interceptor`
    - `intercept()` 逻辑：
      1. 记录 `val start = System.currentTimeMillis()`
      2. `val resp = chain.proceed(request)`
      3. `val elapsed = System.currentTimeMillis() - start`
      4. `resp.code != 200` → `log.warn { "[$method] failed: code=${resp.code} msg=${resp.msg} (${elapsed}ms)" }`
      5. `resp.code == 200` → `log.info { "[$method] success (${elapsed}ms)" }`
      6. 返回 resp

    **RateLimitInterceptor.kt** (D-08):
    - `class RateLimitInterceptor : Interceptor`
    - 使用 `ConcurrentHashMap<String, Semaphore>` 实现基于 userId 的简单并发限流（Review 反馈#4: 不能为空骨架，Phase 5~10 需要基础保护）
    - 限流逻辑：
      1. 从 `coroutineContext[SessionKey]` 获取当前 userId（已认证请求），生成限流 key
      2. 未认证请求使用请求来源 IP 作为 key（`extractClientIp(request)` — 骨架占位，返回 `"unknown"`）
      3. 使用 `computeIfAbsent` 获取对应 userId 的 `Semaphore(20)` — 每用户最大并发 20 个请求
      4. `semaphore.tryAcquire(100, TimeUnit.MILLISECONDS)` — 若 100ms 内获取不到信号量则限流
      5. 未获取到 → 返回 `Response.newBuilder().setCode(429).setMsg("rate limit exceeded")`
      6. 获取到 → `try { chain.proceed(request) } finally { semaphore.release() }`
    - KDoc 注明 "基于 Semaphore 的每用户并发限流骨架，Phase 11 替换为令牌桶算法"
    - 限流阈值（`Semaphore(20)`）和超时时间（`100ms`）通过常量定义，方便后续配置化
    - 注意：需 import `kotlinx.coroutines.coroutineContext` 和 `java.util.concurrent.Semaphore`
    - ⚠️ 实现时注意：RateLimitInterceptor 使用的 Semaphore 需要 import `java.util.concurrent.Semaphore`；`coroutineContext[SessionKey]` 需要 import `kotlinx.coroutines.coroutineContext`。测试时需要 Mock SessionKey 在 CoroutineContext 中的存在（通过 `withContext` 注入 Mock Session）。

    ```kotlin
    package com.nebula.gateway.interceptor

    import com.nebula.chat.Request
    import com.nebula.chat.Response
    import com.nebula.common.BizCode
    import com.nebula.common.exception.BizException
    import io.github.oshai.kotlinlogging.KotlinLogging

    /**
     * 异常拦截器 — 三态异常映射（D-10）。
     *
     * 作为 Pipeline 链尾包裹 Handler，捕获所有异常并映射为结构化 Response。
     * 不向客户端暴露堆栈细节。
     */
    class ExceptionInterceptor : Interceptor {

        override suspend fun intercept(request: Request, chain: Interceptor.Chain): Response {
            return try {
                chain.proceed(request)
            } catch (e: BizException) {
                // D-10: BizException → 业务状态码
                Response.newBuilder()
                    .setCode(e.bizCode.code)
                    .setMsg(e.message ?: e.bizCode.msg)
                    .setMethod(request.method)
                    .build()
            } catch (e: IllegalArgumentException) {
                // D-10: 参数异常 → BAD_REQUEST
                logger.warn(e) { "Illegal argument for method ${request.method}" }
                Response.newBuilder()
                    .setCode(BizCode.INVALID_PARAM.code)
                    .setMsg(e.message ?: BizCode.INVALID_PARAM.msg)
                    .setMethod(request.method)
                    .build()
            } catch (e: Exception) {
                // D-10: 未预期异常 → INTERNAL_ERROR(9000)，不暴露堆栈
                logger.error(e) { "Unhandled exception for method ${request.method}" }
                Response.newBuilder()
                    .setCode(BizCode.INTERNAL_ERROR.code)
                    .setMsg(BizCode.INTERNAL_ERROR.msg)
                    .setMethod(request.method)
                    .build()
            }
        }

        companion object {
            private val logger = KotlinLogging.logger {}
        }
    }
    ```
    - 注意 catch 顺序：BizException（最具体）→ IllegalArgumentException → Exception（最通用）
    - KDoc 注明三态映射（D-10）
    - 未预期异常：仅 `logger.error` 写堆栈，Response 只返回 9000 + "internal error"

    需要确认 Koin 能解析这些依赖：
    - UserRepository: 在 NebulaServer 中通过 `val userRepo = jpaConfig.getRepository(UserRepository::class.java)` 创建，但不在 Koin 容器中。需要在 frameworkModule 中添加 `single { userRepo }` 等方式。
    
    方案：创建 `repositoryModule` 或在 `frameworkModule` 中注册所有 JPA/Redis Repository 单例：

    ```kotlin
    val frameworkModule = module {
        // ... 现有内容 ...

        // Phase 5: Repository 单例注册（由 NebulaServer 初始化后注入）
        // 注：这些会被 NebulaServer 在 startKoin 前通过 Koin 的 later binding 或
        // 通过 properties 方式注入。更简单的方式是 NebulaServer 在创建 Koin 容器时传入。
    }
    ```

    实际方案：在 `frameworkModule` 中不预注册 Repository。改为在 `NebulaServer.kt` 的 `startKoin { }` 之前调用 `GlobalContext` 或使用 `koin.loadModules()` 动态添加。但最简单的方法是在 `frameworkModule` 中使用 `single { }` 声明空单例，然后在 NebulaServer 中通过 `get()` 获取并使用。

    更好的方案：使用 Koin 的 `single { getProperty<UserRepository>("userRepository") }` 或在 `startKoin` 的 `properties` 参数中传入。

    最简洁的方案：在 `NebulaServer.kt` 的 `startKoin` 之前将所有 Repository 实例化并通过 `single { }` 动态注册：

    实际上，由于 Repository 实例由 JpaConfig 动态创建（`jpaConfig.getRepository(UserRepository::class.java)`），这些实例需要在 Koin 容器外部创建后注入。最直接的做法：

    1. 在 `GatewayModule.kt` 的 `frameworkModule` 中添加类型声明（不在 module 块中—因为它们需要外部创建）：
    相反，在 `NebulaServer.kt` 的 `startKoin` 之前使用 `module { }` 创建：
    
    ```kotlin
    val repoModule = module {
        single { userRepo }
        single { sessionRepo }
        single { onlineStatusRepo }
        // ... 其他 repository
    }
    ```

    但这样耦合太强。更合理的方式：

    修改 `GatewayModule.kt`，在 `frameworkModule` 中注册一个 `UserRepository` 的声明（使用 `single { }` 但 deferred），然后使用 Koin 的 `allowOverride=true` 或使用 `declare` 函数。

    最实际的方案：
    在 `frameworkModule` 中添加：
    ```kotlin
    // Phase 5: 外部 Repository 将在 NebulaServer 启动时通过 Koin properties 或 declared 方式注入
    single<UserRepository> { getProperty("userRepository") }
    single<SessionRepository> { getProperty("sessionRepository") }
    single<OnlineStatusRepository> { getProperty("onlineStatusRepository") }
    ```

    但在 `NebulaServer.kt` 中，在 `startKoin` 时传入 properties：
    ```kotlin
    startKoin {
        properties(mapOf(
            "userRepository" to userRepo,
            "sessionRepository" to sessionRepo,
            "onlineStatusRepository" to onlineStatusRepo,
        ))
        modules(frameworkModule, handlerModule)
    }
    ```

    然后 Handler 中通过 `get()` 获取这些 repository。但是 `getProperty()` 不是类型安全的。

    更简洁的方案：直接在 `NebulaServer.kt` 中把 Repository 注册为 module：
    ```kotlin
    val externalModule = module {
        single { userRepo as UserRepository }
        single { sessionRepo as SessionRepository }
        single { onlineStatusRepo as OnlineStatusRepository }
    }
    startKoin {
        modules(frameworkModule, handlerModule, externalModule)
    }
    ```

    好，就用这个方案。需要在 `GatewayModule.kt` 的 `frameworkModule` 中移除直接 `single { SessionRegistry(get()) }`（因为 SessionRepository 现在由外部 module 提供）。

    等等，让我重新审视：`frameworkModule` 中已经有了 `single { SessionRegistry(get()) }` 和 `single<Interceptor> { AuthInterceptor(get(), ...) }`。这些已经在 Phase 4 工作。`SessionRegistry(get())` 通过 Koin 解析 `SessionRepository`。所以如果我们在外部 module 中注册 `SessionRepository`，SessionRegistry 就能正常解析。


    函数体：为每个 Handler 构建 HandlerEntry 并注册到 registry。注册模式：
    ```kotlin
    // LoginHandler: method="user/login", Req=LoginReq, Resp=LoginResp
    val loginReqCodec = ProtoCodec.buildCodec(LoginReq::class)
    val loginRespCodec = ProtoCodec.buildCodec(LoginResp::class)
    registry.register(HandlerEntry(
        handler = loginHandler,
        reqClass = LoginReq::class,
        respClass = LoginResp::class,
        parseFrom = loginReqCodec.parseFrom,
        toByteArray = loginRespCodec.toByteArray
    ))
    ```

# Plan 14-02: 认证加强（GC5 deviceId 重连验证）+ 依赖清理（S8）

## 目标

加强 Token 重连时的安全性：在 LoginHandler 重连分支中增加 deviceId 验证。
同时清理 server 模块的冗余依赖（Phase 12 后暴露）。

---

## Plan 14-02-A: GC5 deviceId 验证（LoginHandler 重连分支）

### 背景

GC5（D-14-06）要求：Token 重连时验证请求中的 deviceId 与 Session 绑定的 deviceId 是否一致，防止 Token 被跨设备盗用。

Session 已有 `deviceId` 字段（Session.kt:22），无需新增存储字段。
重连请求的 `LoginReq.deviceId` 由客户端传入（protobuf 字段）。

### 任务

| # | 类型 | 文件 | 操作 | 验证 | 验收标准 |
|---|------|------|------|------|---------|
| 1 | modify | `gateway/src/main/kotlin/com/nebula/gateway/handler/user/LoginHandler.kt` | 在 `handle()` 方法的 Token 重连分支（`req.hasToken()` → `existingSession` 非空）中，增加 deviceId 校验：提取 `req.deviceId`，若非空且与 `existingSession.deviceId` 不匹配，抛 `UserException(BizCode.TOKEN_INVALID, "Token device id mismatch")`。`req.deviceId` 为空时跳过验证（兼容旧客户端） | `./gradlew :gateway:compileKotlin` 通过 | deviceId 不匹配时返回 TOKEN_INVALID(1101)；无 deviceId 时兼容通过 |
| 2 | modify | `gateway/src/test/kotlin/com/nebula/gateway/handler/user/LoginHandlerTest.kt` | 新增三个测试方法：`token 重连时 deviceId 不匹配应拒绝`（不同 deviceId → TOKEN_INVALID）、`token 重连时无 deviceId 应兼容`（空 deviceId → 通过）、`token 重连时 deviceId 匹配应通过`（相同 deviceId → 通过，防止误伤正常重连） | 测试通过 | 三个新增测试全部通过 |

---

## Plan 14-02-B: S8 server 模块依赖清理

### 背景

Phase 12 已将 repository/service 模块从 `implementation` 降为 `testImplementation`，消除了 server→repository 直接依赖。因此 Redis/JPA/Spring 等持久化依赖不再需要。

根据 RESEARCH.md 确认，以下 5 个依赖在 server 源码中无直接 import，可安全删除：

| 依赖 | 原因 |
|------|------|
| `libs.lettuce.core` | server 不直接操作 Redis |
| `libs.hibernate.core` | server 不直接使用 JPA |
| `libs.spring.tx` | server 不直接管理事务 |
| `libs.hikaricp` | server 不直接管理连接池 |
| `libs.spring.data.jpa` | server 不直接使用 Spring Data JPA |

`kotlinx.coroutines.core` 通过 gateway 传递，但保留显式声明是标准做法，**不删除**。

### 任务

| # | 类型 | 文件 | 操作 | 验证 | 验收标准 |
|---|------|------|------|------|---------|
| 3 | modify | `server/build.gradle.kts` | 删除 `libs.lettuce.core`、`libs.hibernate.core`、`libs.spring.tx`、`libs.hikaricp`、`libs.spring.data.jpa` 共 5 行 `implementation` 声明 | `./gradlew :server:compileKotlin` 通过 | server 模块编译通过，无缺失依赖 |

# Plan 14-01: Testcontainers Redis 测试基础设施

## 目标

搭建嵌入式 Redis 测试基础设施（Testcontainers），创建复用测试基类，补全 T04/T05/T06 延期测试。

## 背景

主 Redis 基础设施已到位（Lettuce + Redis Stream + 会话缓存），但**测试基础设施缺失**。service 模块当前的测试全部基于 MockK，无 Testcontainers Redis 支持。

### Wave 1-B: T04 memberCount 并发测试（MockK 方案，无需 Redis）

**策略说明**: T04 使用 MockK 模拟 Repository 层，验证 `ConversationService` 在协程并发场景下的调用逻辑正确性。此方案**仅覆盖协程并发调度逻辑**，不验证 MySQL JPA 层面的 `member_count` 原子一致性。真正的 MySQL 级并发一致性测试需 MySQL Testcontainers 环境，涉及 EntityManager 事务边界，超出本阶段 Redis 基础设施范围，建议在后续 MySQL 测试专项中完成。

**局限性**: 所有 Repository 调用被 MockK 替换后，`coroutineScope { launch { ... } }` 下的调用实际是顺序执行 MockK 预配置行为，无法验证 MySQL JPA 层的 `member_count` 原子性。此测试**不替代** MySQL 级并发测试，仅为协程调度逻辑的轻量验证。

| # | 类型 | 文件 | 操作 | 验证 | 验收标准 |
|---|------|------|------|------|---------|
| 4 | test | `service/src/test/kotlin/com/nebula/service/conversation/ConversationServiceTest.kt` | 新增测试方法 `memberCount 并发更新应保持协程一致性`：使用 `coroutineScope { repeat(5) { launch { conversationService.inviteMember(convId, userId) }; launch { conversationService.leaveGroup(convId, userId) } } }` 并发执行，验证最终 `conversation.memberCount == countActiveMembers()`。**注意**: 本测试使用 MockK，仅覆盖协程并发层面的调用序列正确性 | `./gradlew :service:test --tests "*ConversationServiceTest*memberCount*"` 通过 | 并发 invite/leave 后 member_count 与 Mock 返回值一致 |
